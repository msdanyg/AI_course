The Reasoning Engine: Advanced Prompt Architecture

Learn to structure prompts for consistent, high-quality results

Key Insight

It's not about XML specifically—it's about clear separation of concerns. You can achieve this with XML tags, Markdown headers, or even simple labels. Choose what feels natural to you.

What You'll Learn

  • Why unstructured prompts produce inconsistent results ("context bleeding")
  • Three format options: XML tags, Markdown headers, or clear section labels
  • When XML is superior (and when simpler formats work fine)
  • How to ask Claude to structure messy data for you
  • When you don't need formal structure at all

The Structure Spectrum

You have three options for structuring prompts. All three work—choose based on comfort level and task complexity:

Compare Format Options
Most Precise. Best for: complex prompts, nested data, reusable templates, structured outputs.
<system_role> You are a CSM at ActivTrak focused on "Insights, Not Oversight." </system_role> <task_context> Client productivity dropped 15% during EHR implementation. </task_context> <constraints> - Avoid: tracking, monitoring, surveillance - Use supportive framing </constraints> <output_format> Three hypotheses with supporting data. </output_format>
Balanced. Best for: narrative prompts, documentation-style, people uncomfortable with XML.
## Role You are a CSM at ActivTrak focused on "Insights, Not Oversight." ## Context Client productivity dropped 15% during EHR implementation. ## Constraints - Avoid: tracking, monitoring, surveillance - Use supportive framing ## What I Need Three hypotheses with supporting data.
Simplest. Best for: quick tasks, beginners, when formal structure feels like overkill.
ROLE: CSM at ActivTrak, "Insights, Not Oversight" SITUATION: Client productivity dropped 15% during EHR implementation AVOID: tracking, monitoring, surveillance language TASK: Three hypotheses with supporting data

When to Use Structure (And When Not To)

✓ Use Formal Structure When

  • • Multiple data sources
  • • Several distinct requirements
  • • Getting inconsistent results
  • • Creating reusable templates

✗ Skip Structure When

  • • Simple, single-purpose task
  • • Back-and-forth conversation
  • • Your writing is already clear
  • • Quick questions

Pro Tip: Let Claude Structure Your Data

Have messy meeting notes or raw data? Just ask Claude to organize it:

"Here are my rough notes. Convert them into structured sections: Client Goals, Concerns, Action Items, Next Steps."

Then use Claude's structured output as clean input for your next prompt.

The Reasoning Engine: Advanced Prompt Architecture

Estimated reading time: 15-20 minutes

The Core Principle

Clear structure prevents context bleeding. When you separate role, data, constraints and instructions—whether using XML tags, Markdown headers, or simple labels—the AI produces consistent, high-quality outputs.

The goal is clarity, not a specific syntax. Choose the format that feels natural to you.

When XML Is Superior

While all formats work, XML has advantages in specific situations:

Situation Why XML Works Best
Nested data Tags inside tags (metrics within client_info within user_data)
Structured output Signals you want data back in specific format
Reusable templates Clear placeholders like [CLIENT_NAME]
Markdown in data Prevents confusion when your data contains ## headers

Standard Components

Regardless of format, include these components:

Component Purpose
Role Who the AI should be
Context Background situation
Data Content to process
Constraints What NOT to do
Output Format Desired structure

Order matters: Role at top, data in middle, instructions at end (due to "Lost in the Middle" effect).

Common Mistakes

Mistake Fix
Overloading (research + write + critique) Separate tasks across conversations
Vague outputs ("make it better") Specify exactly what "better" means
Missing constraints Always include what to AVOID
Buried instructions Place key instructions at END
Over-engineering Match structure to task complexity

Study Guide

Quick reference for Module 3 concepts

Core Concept

Clear structure prevents context bleeding. The goal is clarity, not a specific syntax. Choose XML, Markdown, or simple labels based on comfort and task complexity.

The Structure Spectrum

Format Best For
XML Tags Complex prompts, nested data, templates, structured outputs
Markdown Headers Narrative prompts, documentation-style, XML-uncomfortable users
Clear Sections Simple tasks, beginners, when formal structure feels like overkill

When XML Is Superior

  • Nested data (tags inside tags)
  • Requesting structured output
  • Reusable templates with placeholders
  • When your data contains Markdown formatting

Pro Tip

Ask Claude to structure messy data: "Here are my rough notes. Convert them into sections: Goals, Concerns, Action Items, Next Steps."
