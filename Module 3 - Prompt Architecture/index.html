The Reasoning Engine: Advanced Prompt Architecture

Learn to structure prompts for consistent, high-quality results

Key Insight

It's not about XML specifically—it's about clear separation of concerns. You can achieve this with XML tags, Markdown headers, or even simple labels. Choose what feels natural to you.

What You'll Learn

  • Why unstructured prompts produce inconsistent results ("context bleeding")
  • Three format options: XML tags, Markdown headers, or clear section labels
  • When XML is superior (and when simpler formats work fine)
  • How to ask Claude to structure messy data for you
  • When you don't need formal structure at all

The Structure Spectrum

You have three options for structuring prompts, progressing from simplest to most advanced. All three work—choose based on comfort level and task complexity:

Compare Format Options
Simplest Approach. Best for: beginners, quick tasks, when formal structure feels like overkill.
ROLE: CSM at ActivTrak, "Insights, Not Oversight" SITUATION: Client productivity dropped 15% during EHR implementation AVOID: tracking, monitoring, surveillance language TASK: Three hypotheses with supporting data
Balanced. Best for: narrative prompts, documentation-style, people comfortable with Markdown.
## Role You are a CSM at ActivTrak focused on "Insights, Not Oversight." ## Context Client productivity dropped 15% during EHR implementation. ## Constraints - Avoid: tracking, monitoring, surveillance - Use supportive framing ## What I Need Three hypotheses with supporting data.
Most Precise. Best for: complex prompts, nested data, reusable templates, structured outputs, advanced users.
<system_role> You are a CSM at ActivTrak focused on "Insights, Not Oversight." </system_role> <task_context> Client productivity dropped 15% during EHR implementation. </task_context> <constraints> - Avoid: tracking, monitoring, surveillance - Use supportive framing </constraints> <output_format> Three hypotheses with supporting data. </output_format>

Markdown Quick Reference

If you're new to Markdown, here's a quick reference for the most useful formatting you'll use in prompts:

Headers

# Level 1 Header ## Level 2 Header ### Level 3 Header

Text Formatting

**Bold text** *Italic text* ***Bold and italic***

Lists

Bullet list: - Item one - Item two - Nested item Numbered list: 1. First item 2. Second item

Code

Inline: `variable_name` Code block: ``` code here ```

Tables

| Column 1 | Column 2 | |----------|----------| | Data 1 | Data 2 |

Links & Rules

[Link text](url) Horizontal rule: ---

Why this matters for prompting: Markdown formatting helps you organize data within prompts (tables for metrics, lists for findings, code blocks for technical content) and helps Claude parse structured information more reliably.

When to Use Structure (And When Not To)

✓ Use Formal Structure When

  • • Multiple data sources
  • • Several distinct requirements
  • • Getting inconsistent results
  • • Creating reusable templates

✗ Skip Structure When

  • • Simple, single-purpose task
  • • Back-and-forth conversation
  • • Your writing is already clear
  • • Quick questions

Pro Tip: Let Claude Structure Your Data

Have messy meeting notes or raw data? Just ask Claude to organize it:

"Here are my rough notes. Convert them into structured sections: Client Goals, Concerns, Action Items, Next Steps."

Then use Claude's structured output as clean input for your next prompt.

The Reasoning Engine: Advanced Prompt Architecture

Estimated reading time: 15-20 minutes

The Core Principle

Clear structure prevents context bleeding. When you separate role, data, constraints and instructions—whether using XML tags, Markdown headers, or simple labels—the AI produces consistent, high-quality outputs.

The goal is clarity, not a specific syntax. Choose the format that feels natural to you.

When XML Is Superior

While all formats work, XML has advantages in specific situations:

Situation Why XML Works Best
Nested data Tags inside tags (metrics within client_info within user_data)
Structured output Signals you want data back in specific format
Reusable templates Clear placeholders like [CLIENT_NAME]
Markdown in data Prevents confusion when your data contains ## headers

Standard Components

Regardless of format, include these components:

Component Purpose
Role Who the AI should be
Context Background situation
Data Content to process
Constraints What NOT to do
Output Format Desired structure

Order matters: Role at top, data in middle, instructions at end (due to "Lost in the Middle" effect).

Common Mistakes

Mistake Fix
Overloading (research + write + critique) Separate tasks across conversations
Vague outputs ("make it better") Specify exactly what "better" means
Missing constraints Always include what to AVOID
Buried instructions Place key instructions at END
Over-engineering Match structure to task complexity

Part A: Email to Template (10 minutes)

The Assignment

Your Goal: Convert the Forever New POC closeout email into a reusable markdown template with placeholders.

Data Source: The POC Closeout Email

📧 View POC Closeout Email (Click to expand)
Subject: Forever New POC Results - Ready to Discuss Next Steps Hi Ben and team, I wanted to share the results from your ActivTrak POC. The data is really compelling. Your team showed 0.3 hours more productivity when working remotely (6.8 hrs) compared to in-office (6.5 hrs). The Buying Apparel team especially benefited from remote work. Interestingly, hybrid days where people split locations had the lowest productivity at 6.4 hours - likely due to transition time. We also identified significant capacity opportunities. 68.6% of users (105 of 153) are currently below productivity goals, representing about $104K in monthly financial loss or 25 FTE worth of untapped capacity. The IT Department spends 27% of their time in chat and messaging - nearly double other departments - suggesting focus time optimization opportunities. We detected 3 users with potential time theft issues using input simulation tools, which validates your concerns about activity authenticity. Based on these findings, I recommend we focus on three immediate actions: addressing the IT productivity gap, coaching the bottom 36 users averaging under 5 productive hours daily, and investigating the input simulation cases. The ROI potential is substantial - even a conservative 10% improvement would recover $125K annually and gain 2.5 FTE capacity without hiring. I'd love to schedule a call this week to discuss moving forward with Essentials Plus. Are you available Thursday or Friday afternoon? Best regards, Sarah Mitchell Customer Success Manager, ActivTrak

Your Instructions

Create a prompt that tells Claude to:

  1. Replace specific details with placeholders in [BRACKETS]
  2. Organize into clear sections with markdown headers (##)
  3. Preserve the tone and flow but make it generic enough for any POC
  4. Make it flexible - some POCs might have 2 findings, others might have 5

Try It Yourself First

Open Claude and write your own prompt. See if you can get it to produce a good template!

💡 Show Me How (Click to reveal example prompt)

Here's an example using the ActivTrak standard prompt components:

ROLE: You are a Solutions Consultant at ActivTrak creating reusable templates for POC closeout communications. CONTEXT: This template will be used by the Solutions team for future POC closeouts with different clients. It needs to be flexible enough to work for various findings while maintaining a consistent professional structure. DATA: [Paste the POC closeout email above] CONSTRAINTS: - Replace all client-specific details with placeholders in [BRACKETS] - Use descriptive placeholder names (e.g., [CLIENT_NAME], [PRODUCTIVITY_METRIC], [ROI_FIGURE]) - Preserve the supportive, consultative tone - Keep the email structure flexible for 2-5 key findings - Maintain professional closing format OUTPUT FORMAT: Create a markdown template with these sections: ## Subject Line ## Greeting ## Key Findings (flexible number) ## Recommendations ## ROI Summary ## Next Steps ## Closing

What makes this work:

  • Role establishes who Claude is and their purpose
  • Context explains why this template matters and how it will be used
  • Data clearly separates the source material
  • Constraints prevent common errors (vague placeholders, rigid structure)
  • Output Format specifies exactly what sections to create

Part B: Multi-Asset Executive Summary (10-15 minutes)

The Assignment

Your Goal: Create a structured prompt that synthesizes THREE data sources into a board-ready executive summary for Forever New's CIO.

The Challenge: How do you structure a prompt with multiple data sources without context bleeding?

Your Instructions

Structure a prompt that includes:

  1. Role: Who is Claude? (CSM preparing executive summary for C-suite)
  2. Context: What's the purpose and audience? (Board-ready POC results, non-technical executives)
  3. Data Sources: All three sources - clearly separated
  4. Constraints:
    • Use "users" or "accounts" (never "customers" for end users)
    • Use "analytics" and "insights" (never "monitoring" or "tracking")
    • Frame findings supportively using "Insights, Not Oversight" philosophy
    • No technical jargon - audience is C-suite
    • Keep summary to 400 words maximum
  5. Output Format: POC Overview, Key Findings, Strategic Recommendations, Expected ROI, Proposed Next Steps

Data Sources

You'll need all three data sources to complete Part B. Notice that each source uses a different format (free text, Markdown, bullet list) - this is intentional! Real-world data rarely comes in a single format, and your prompt structure must handle this variety without context bleeding.

Bonus Challenge: One Prompt, Two Outputs (Advanced)

The Assignment

Can you create a single structured prompt that produces BOTH:

  1. The reusable email template (Part A)
  2. The executive summary (Part B)

Why this is hard: Different output formats, different audiences, risk of context bleeding between template creation and summary writing.

Hint: You'll need nested structure - perhaps a parent <task> tag with two child <deliverable> tags, each with their own constraints and output formats.

💡 Show Me How (Click to reveal)

Here's one way to structure a combined prompt using nested XML:

<task_overview> You are a Customer Success Manager at ActivTrak. You need to create TWO deliverables from POC closeout materials. </task_overview> <deliverable_1> <type>Reusable Template</type> <goal>Convert the POC email into a markdown template for future use</goal> <instructions>Replace specific details with [PLACEHOLDERS], organize with ## headers</instructions> </deliverable_1> <deliverable_2> <type>Executive Summary</type> <goal>Create board-ready summary synthesizing all three data sources</goal> <constraints>ActivTrak terminology, 400 words, no surveillance language</constraints> <output_format>5 required sections (POC Overview, Findings, etc.)</output_format> </deliverable_2>

Key insight: Nested tags prevent context bleeding between the two tasks.

Quick reference for Module 3 concepts

Core Concept

Clear structure prevents context bleeding. The goal is clarity, not a specific syntax. Choose XML, Markdown, or simple labels based on comfort and task complexity.

The Structure Spectrum

Format Best For
XML Tags Complex prompts, nested data, templates, structured outputs
Markdown Headers Narrative prompts, documentation-style, XML-uncomfortable users
Clear Sections Simple tasks, beginners, when formal structure feels like overkill

When XML Is Superior

  • Nested data (tags inside tags)
  • Requesting structured output
  • Reusable templates with placeholders
  • When your data contains Markdown formatting

Pro Tip

Ask Claude to structure messy data: "Here are my rough notes. Convert them into sections: Goals, Concerns, Action Items, Next Steps."
